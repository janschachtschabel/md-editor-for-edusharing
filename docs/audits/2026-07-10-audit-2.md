## 1. Executive Summary

Die heutigen Feature-Runden sind sauber getestet gelandet (**277 Checks, 0
Fails**, 0 CVEs, Lint 0, keine Zirkel) — Logout-Härtung und das neue
Keyword-Semantikmodell sind konzeptionell richtig und hook-getestet. **Aber der
Audit hat im neuen Save-Anker-Filter einen bewiesenen Korrektheitsfehler
gefunden (KW-1):** Der Server prüft die Zitat-Verankerung gegen den
**Markdown-Quelltext**, der Client gegen den **Plain-Text** des Editors. Folge:
Ein Tag, dessen Zitat eine Formatierungsgrenze überspannt (`**Kartoffelhof**
Sonnental`) oder Escape-Zeichen enthält (`snake\_case\_name` — Turndown
escapt Unterstriche!), gilt serverseitig als unverankert und wird beim
nächsten Speichern **still gelöscht — ohne dass die Pille je grau wurde**
(der Client-Anker hält ja). Das ist stiller Tag-Verlust durch normale
Formatierung. **Wichtigste Maßnahme:** Server-Anker auf Plain-Text umstellen,
bevor das Tagging produktiv genutzt wird.

## 3. Findings

### KW-1 🟠 High — Save-Anker prüft Markdown-Quelltext, Client Plain-Text → stiller Tag-Verlust
```
File:    server/collab.js (persistDocument: textbase-Filter + Prune)
         vs. src/annotation-extension.js (buildTextIndex: Plain-Text-Anker)
Problem: Der neue Anker-Filter/-Prune nutzt findQuoteRange(markdown + otherText,
         quote) — also den SERIALISIERTEN Markdown. Der Client (Pillen-Status,
         Decorations) ankert gegen den Plain-Text des Editors.
Beweis:  (reproduziert) quote 'Kartoffelhof Sonnental' bei fettem 'Kartoffelhof'
         → markdown 'Der **Kartoffelhof** Sonnental …' → Server-Anker null,
         Client-Anker {4,26}. Ebenso 'snake_case_name' → markdown
         'snake\_case\_name' (Turndown-Escaping) → Server-Anker null.
Impact:  Normale Redaktionsarbeit (Wort im getaggten Bereich fetten; Begriffe
         mit _ * # etc.) löscht Tags beim nächsten Speichern still — die Pille
         war bis dahin NORMAL blau (keine Orphan-Warnung), verschwindet dann
         kommentarlos samt Keyword.
Fix:     Server-Verankerung auf Plain-Text umstellen: (a) aktuelles Dokument —
         Plain-Text direkt aus dem Y.Doc-Fragment extrahieren (Blöcke mit \n
         getrennt, analog buildTextIndex/ytextOf); (b) otherText — Markdown
         des anderen Felds zu Plain wandeln (markdownToHtml → Tags strippen
         oder generateJSON → Text-Walker). Zusätzlich Regressionstests:
         Zitat über Bold-Grenze + Zitat mit Escape-Zeichen überlebt Save.
```

### UX-1 🟡 Medium — `authenticationFailed` unbehandelt: Session-Ablauf/Logout-Kick ohne Nutzer-Feedback
```
File:    src/md-collab-editor.js (Provider-Setup), src/host.js
Problem: Seit ungültige Tokens abgelehnt werden (Logout-Härtung, korrekt),
         erhält ein Client mit abgelaufener (8 h) oder widerrufener Session
         beim (Re-)Connect ein authenticationFailed — weder Komponente noch
         Host hören darauf (grep: keine Treffer). Der Provider versucht
         zudem periodisch neu zu verbinden (Ablehnungs-Schleife im Serverlog).
Impact:  Nutzer sieht nur „getrennt" ohne Grund; Server loggt wiederholte
         Ablehnungen desselben Clients.
Fix:     onAuthenticationFailed im Provider-Setup: provider.disconnect()
         (Reconnect-Schleife stoppen) + neues status-Event 'session-expired'
         → Host zeigt „Sitzung abgelaufen — bitte neu anmelden".
```
