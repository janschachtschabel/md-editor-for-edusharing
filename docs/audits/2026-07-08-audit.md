# Codebase Audit — Collaborative Markdown Editor for edu-sharing

**Date:** 2026-07-08
**Depth:** Standard (all 12 dimensions; line-by-line read of every server module
and every `src/` module; scanners run: `npm audit`, `npm outdated`, `npm test`,
`npm run lint`, `npm run build`)
**Auditor role:** multi-lens read-only review (Architect, Security, Performance,
API, Frontend/UX, QA, DevOps, Dependency Manager) — no code was modified.

---

## 1. Executive summary

The application is a small (~3,500 LOC), well-engineered Node.js/Express +
Hocuspocus/Yjs collaborative markdown editor for edu-sharing, with unusually
mature security hygiene for its size — most classic web risks (path traversal
into the upstream API, IDOR on write access, XSS via `innerHTML`, secrets in
the repo, missing rate limiting on login, CSRF) are already addressed and
regression-tested, with in-code references to prior audit findings ("audit
F-01" etc.), indicating this is not the first review cycle. The main risk is a
**single but real authentication bypass**: the Basic-auth passthrough path
(`Authorization: Basic <base64>` sent directly to `/api/nodes/:id`,
`/api/nodes/:id/save`, `/api/nodes/:id/autosave`) is **not covered by the
login rate limiter**, turning this server into an unthrottled credential-
guessing oracle against the upstream edu-sharing repository. Secondary risks:
zero automated test coverage for the two largest and most interaction-heavy
modules (`md-collab-editor.js`, `host.js`, `annotation-ui.js` — all DOM/UI
glue code, together ~950 lines), and documentation that already drifted from
the code within this session (README test-suite count, missing `lang`
attribute). None of the findings are exploitable-today critical vulnerabilities;
the codebase is close to production-ready conditional on closing the
rate-limiting gap.

**Top 3 risks:**
1. Basic-auth passthrough bypasses the login rate limiter → brute-force oracle against edu-sharing credentials (Security).
2. No automated tests for the DOM-facing modules that implement the annotation UI, save bar wiring, and host page — the most frequently touched code in recent sessions (Testing).
3. Documentation drift (README test-suite count, missing `lang` attribute, `.gitlab-ci.yml`/`ci.yml` comments) undermines trust in the docs as onboarding material (Documentation/Maintainability).

**Single most important action:** apply the login rate limiter (or an
equivalent per-IP throttle) to every endpoint that accepts a raw `Basic`
Authorization header, not just `/api/login`.

## 2. Overview

- **Stack:** Node.js 22, Express 4, Hocuspocus 4 (Yjs WebSocket server),
  TipTap 3 (ProseMirror), Yjs 13, marked/turndown for Markdown⇄HTML, vanilla
  JS Web Component (`<md-collab-editor>`) — no frontend framework, no
  database (edu-sharing REST API is the system of record).
- **Purpose:** collaborative real-time editing of "kompendiale Texte" on
  edu-sharing nodes, with a standoff semantic-tagging layer persisted as
  `cclom:general_keyword`.
- **Size:** 12 `src/` modules, 5 `server/` modules, 8 test suites, ~3,500 LOC
  excluding `node_modules`, docs, and generated bundles.
- **Coverage of this audit:** every file in `server/`, every file in `src/`,
  `server.js`, `package.json`, `Dockerfile`, `docker-compose.yml`,
  `.github/workflows/*.yml`, `.gitlab-ci.yml`, `eslint.config.js`,
  `.env.example`, `.gitignore`, all 8 test suites, `public/index.html`,
  `public/app-config.js`, and the first ~220 lines of `README.md`, were read
  in full. `npm audit`, `npm outdated`, `npm test`, `npm run lint`, and
  `npm run build` were executed and their output verified.
- **Not read in full (skimmed or skipped):** `README.en.md`,
  `docs/ENTWICKLERLEITFADEN.md`, `docs/SEMANTISCHES-TAGGING.md`,
  `docs/SEMANTISCHES-TAGGING-INTEGRATION.md`, `THIRD-PARTY-NOTICES.md`,
  `public/style.css` (438 lines, spot-checked only), `mds.json` (17 MB,
  gitignored scratch artifact — out of scope). No live penetration testing,
  no load testing, no dependency license audit beyond what `README.md`
  already asserts.

## 4. Verdict

**Conditional.** The application is close to production-ready for its stated
scope (a demo/reference embedding for edu-sharing), but should not go into a
production deployment with real user credentials until:

1. The Basic-auth passthrough is rate-limited the same way `/api/login` is (S-1) — this is the one finding with a direct, exploitable, evidenced impact on a third-party system's account security.
2. At least smoke-level tests exist for the annotation dialogs and the save-bar wiring in `md-collab-editor.js` (T-1), since this is the code most recently and most frequently changed.
3. README/CI comments are re-synced to the actual test-suite count and attribute list (D-1..D-3) — low effort, but currently misleading for new contributors.

No other finding in this audit blocks a production decision by itself.

## 5. Top issues

```
[HIGH] Security — Basic-auth passthrough bypasses the login rate limiter
  File:     server/sessions.js, server.js
  Lines:    sessions.js 63-70 (resolveAuthToken); server.js 131-152, 159-183 (route handlers)
  Problem:  resolveAuthToken() accepts any raw "Basic <base64>" Authorization
            header and passes it straight through to edu-sharing on
            GET /api/nodes/:id, POST /api/nodes/:id/save and
            POST /api/nodes/:id/autosave. None of these routes are covered by
            `loginLimiter`, which only wraps POST /api/login.
  Why:      An attacker can send arbitrary Basic-auth guesses directly to
            these routes and learn (via the resulting 200/401/403 status)
            whether the credentials were valid, with zero throttling.
  Impact:   This server becomes an unthrottled proxy for brute-forcing
            edu-sharing accounts — the exact class of attack the login rate
            limiter was built to prevent (see the F-05 comment in
            server.js:85-86), just reachable through a different door.
  Fix:      Apply the same (or a dedicated, stricter) rate limiter — keyed by
            client IP — to any request whose Authorization header is a raw
            "Basic ..." value, before it reaches checkWriteAccess/getNodeInfo.
            Alternatively, reject raw Basic auth on these routes entirely and
            require callers to obtain a session token via /api/login first
            (which is already rate-limited).
  Example:  In server.js, wrap resolveAuthToken(...) results: if
            authHeader?.startsWith('Basic ') && !loginLimiter(req.ip) →
            respond 429 before calling getNodeInfo/checkWriteAccess.
```

```
[MEDIUM] Testing — Zero test coverage on the DOM-facing annotation/editor/host modules
  File:     src/md-collab-editor.js, src/host.js, src/annotation-ui.js, src/annotation-controller.js
  Lines:    whole files (448 + 337 + 162 + 160 lines)
  Problem:  All 8 test suites exercise pure logic (annotations.js,
            entity-types.js, save-state.js, i18n.js) or server HTTP behavior.
            None exercise the web component, the tag/manage dialogs, the
            save-bar button wiring, or host.js's login/logout/document-open
            flow.
  Why:      These are exactly the files most frequently touched across the
            last several sessions (per project history: duplication fix,
            logout fix, i18n rollout) — behavior changes here currently rely
            entirely on manual browser verification.
  Impact:   A regression in, e.g., the crossing-rule error surfacing in
            openTagDialog, or the logout→closeDocument flow, would only be
            caught by manual testing, not CI.
  Fix:      Add a lightweight DOM test harness (jsdom or Playwright component
            tests) covering at minimum: AnnotationController._push crossing
            rejection end-to-end through openTagDialog, and host.js's
            logout → closeDocument → editor-slot empty assertion.
  Example:  `import { JSDOM } from 'jsdom'` in a new test/annotation-ui.test.mjs,
            constructing a minimal root element and asserting openTagDialog's
            rendered DOM + submit behavior.
```

```
[MEDIUM] Frontend/UX — Tag/manage popups are not accessible dialogs
  File:     src/annotation-ui.js
  Lines:    21-43 (openPopup), 56-92 (openTagDialog), 98-115 (openManageDialog)
  Problem:  Popups get `role="dialog"` but no `aria-modal="true"`, no focus
            trap (Tab can leave the popup into the underlying editor), and no
            focus restoration to the triggering element on close.
  Why:      Screen-reader and keyboard-only users lose context: focus can
            wander behind the dialog while it's still open, and after closing
            it, focus is dropped to <body> instead of returning to the
            "🏷 Entität" button or the clicked span.
  Impact:   WCAG 2.2 AA (2.4.3 Focus Order, 4.1.2 Name/Role/Value for modal
            dialogs) is not met for this interaction.
  Fix:      Add `aria-modal="true"`, trap Tab/Shift+Tab within `el`, store
            `document.activeElement` before opening and call `.focus()` on it
            in `close()`.
```

```
[LOW-MEDIUM] Performance — Full-document text index rebuilt on every transaction
  File:     src/annotation-extension.js
  Lines:    28-58 (buildTextIndex), 80-89 (buildDecorations), 116-121 (plugin apply)
  Problem:  Every ProseMirror transaction with `tr.docChanged` walks the
            entire document (`doc.descendants`) to rebuild the text index,
            then re-scans it for every annotation's every occurrence
            (`findAllQuoteRanges`, itself O(n) per call) to rebuild all
            decorations.
  Why:      This is O(document length × annotation count) per keystroke that
            changes the document, with no memoization across transactions.
  Impact:   Fine for compendium-length texts (hundreds to a few thousand
            characters) and typical annotation counts; would visibly degrade
            typing latency on much larger documents or documents with many
            (dozens+) tagged entities.
  Fix:      If larger documents become a real use case, memoize buildTextIndex
            per document version, or restrict decoration rebuilds to the
            changed range via `tr.mapping`.
```

  Lines:    README.md:108-111; .gitlab-ci.yml:37-39; ci.yml:52-55

```
[LOW] Logic — No length bound on tagged quotes / keyword values
  File:     src/annotations.js, server/edu-sharing-api.js
  Lines:    annotations.js 28-31 (isValidType), 34-36 (formatKeyword); edu-sharing-api.js 159-171 (setProperty/saveKeywords)
  Problem:  A user can select an arbitrarily long single-line span (no `\n`)
            as a "quote" and tag it; formatKeyword concatenates it verbatim
            into a general keyword with no length cap before it is sent to
            edu-sharing via setProperty.
  Why:      edu-sharing (or its underlying Alfresco/Solr layer) may have
            practical limits on individual keyword/property value length that
            are not checked client- or server-side here.
  Impact:   A very long selection tagged as an entity could produce a save
            error surfaced only after a round trip to edu-sharing (via the
            existing read-back verification), rather than being caught early
            with a clear message.
  Fix:      Add a reasonable max-length check in isValidType or _push (e.g.
            reject/warn above ~200 characters) with a translated error
            message via the existing i18n infrastructure.

## 6. All findings (grouped by dimension, then severity)

### Security
- 🟠 **S-1** [HIGH] Basic-auth passthrough bypasses the login rate limiter — see Top Issues.
- 🟢 **S-2** [LOW] `ALLOW_ANONYMOUS_EDIT=true` and `ALLOWED_ORIGINS=*` are both valid configuration values documented as "local only" / "any site" respectively (`.env.example:24, 42`; `server/config.js:53, 61-63`) but nothing at runtime prevents an operator from setting either in a production `.env`/`docker-compose.yml` — the warning is a code comment only, not an enforced guard (e.g. no check against `NODE_ENV=production`). *Needs verification*: whether this is an acceptable trade-off for a project explicitly designed to be embedded/configured by trusted operators; flagged for awareness rather than as a definite defect.

### Logic & Correctness
- 🟢 **L-1** [LOW] No length bound on tagged quotes / keyword values — see Top Issues.
- 🟢 **L-2** [LOW] `occurrenceOfIndex` (`src/annotations.js:83-92`) returns `0` both for "not found" and for a genuine zero-based mismatch; callers (`annotation-controller.js:86`) pass this straight into `occurrence`, and `findQuoteRange` (`annotations.js:54-62`) treats `occurrence: 0` the same as `occurrence: 1` would not be reached (loop `for (let i=0;i<occurrence;i++)` with `occurrence=0` never executes, `start` stays `-1`, returns `null`). This means a selection that ends up not matching any occurrence silently fails to anchor rather than raising a distinguishable error — low likelihood given how it's called (the selection itself must contain the quote), but the ambiguous `0` return is worth a comment or a distinct sentinel.

### Performance
- 🟡 **P-1** [LOW-MEDIUM] Full-document text index rebuilt on every transaction — see Top Issues.
- 🟢 **P-2** [LOW] `server/guards.js:13-31` `createRateLimiter`'s opportunistic sweep only triggers once `hits.size > 10_000`, and the sweep condition (`arr.every((t) => now - t >= windowMs)`) only drops keys whose *entire* history is stale — a key with even one recent hit is never swept, so a slow, sub-limit drip from many distinct IPs could grow the map indefinitely between individual full-staleness sweeps. Bounded in practice by the login endpoint's low traffic; would matter more if reused for a higher-traffic route (relevant if S-1's fix reuses this limiter broadly).

### API
- 🟡 **API-1** [MEDIUM] No rate limiting on `GET /api/nodes/:id` (`server.js:131-152`) at all, independent of the Basic-auth passthrough issue — this endpoint accepts a session Bearer token as well and is polled by the host every `NODE_INFO_POLL_MS` (30s) per open document, but is also reachable by anyone crafting requests directly; combined with S-1 this is the concrete exploitation path.
- 🟢 **API-2** [LOW] Error responses are not uniformly shaped: some return `{error: string}` (most routes), success payloads vary per route (`{token, displayName, authorityName}` vs `{autosave}` vs `{lastSavedAt, lastError, dirty}`) — acceptable for an API this size, but worth a shared envelope if the API surface grows.

### Frontend / UX
- 🟡 **F-1** [MEDIUM] Tag/manage popups are not accessible dialogs — see Top Issues.
- 🟢 **F-2** [LOW] `src/annotation-ui.js:26-27` positions popups with fixed `left`/`top` from the triggering click/selection coordinates and never re-clamps against the viewport — a selection near the right or bottom edge of the window can render the popup partially or fully off-screen.
- 🟢 **F-3** [LOW] The language switcher (`public/index.html:20-23`, `src/host.js:40-43`) reloads the entire page on change; acceptable for this demo host page, but would not carry over an in-progress unsaved edit's local Yjs buffer if the user is mid-session and switches language (the `beforeunload` warning in `md-collab-editor.js:219-224` does not fire for a same-tab `location.reload()` triggered by an explicit user action — actually it does fire since `beforeunload` fires on any navigation including reload; *needs verification* whether the dirty-check timing raises the warning correctly here, since the reload happens synchronously right after `setLang`).

### Testing
- 🟠 **T-1** [MEDIUM] Zero test coverage on `md-collab-editor.js`, `host.js`, `annotation-ui.js`, `annotation-controller.js` — see Top Issues.
- 🟢 **T-2** [LOW] `test/i18n.test.mjs` checks that a curated `SAMPLE_KEYS` list resolves in both languages, but does not assert full parity between the `de` and `en` dictionaries in `src/i18n.js` (i.e. it would not catch a key present in `de` but missing in `en` unless that key happens to be in `SAMPLE_KEYS`). A full key-set diff would be a stronger regression guard, especially since `t()`'s fallback-to-key behavior would silently mask a missing English string as raw dot-path text in production.

### Maintainability
- 🟢 **M-1** [LOW] `src/md-collab-editor.js` (448 lines) inlines toolbar rendering, save-bar rendering, and presence-chip rendering directly as private methods on the same class that also owns the Hocuspocus provider and TipTap editor lifecycle — each concern is small and well-commented, but a new contributor has to read the whole file to understand any one of them. Splitting `_renderToolbar`/`_renderSaveBar`/`_renderUsers` into sibling render modules (mirroring the existing `annotation-ui.js` extraction) would keep the class itself focused on wiring.
- 🟢 **M-2** [LOW] `src/host.js` (337 lines) mixes session storage, DOM wiring, and status-rendering logic in one flat module with no internal sectioning beyond comments — same observation as M-1, lower priority since it's demo/reference code explicitly meant to be replaced by the real Angular host.

### Dependencies
- 🟢 **DEP-1** [LOW] `npm outdated` shows all `@tiptap/*` packages one patch version behind (3.27.1 → 3.27.3) — no known vulnerability, routine update.
- 🟢 **DEP-2** [LOW] Express 4.22.2 → 5.2.1 major version available but not adopted; Express 4 remains maintained, this is a discretionary future migration, not a defect.

### Documentation
- 🟢 **D-1** [LOW] README test-suite count drifted ("7 Suiten" vs actual 8) — see Top Issues.
- 🟢 **D-2** [LOW] CI comments understate the suite count/list ("5 suites") in both `.gitlab-ci.yml` and `.github/workflows/ci.yml` — see Top Issues.
- 🟢 **D-3** [LOW] `lang` attribute undocumented in README's public attribute table — see Top Issues.
- 🟢 **D-4** [LOW] No `CLAUDE.md`/`AGENTS.md` project constitution file exists (only an empty `.claude/` directory); the project relies on `README.md` + `docs/ENTWICKLERLEITFADEN.md` instead, which is a reasonable substitute but means there is no single canonical "rules for AI agents working in this repo" file to check code against, as this audit method would otherwise do in step 4.

### Deployment & Ops
- 🟢 **OPS-1** [LOW] No documented rollback procedure beyond `serverupdate.txt`'s `git pull && docker compose down && docker compose up --build -d` — a failed build/deploy has no documented fast-revert path (e.g. re-tagging the previous image) beyond `git checkout <previous-tag>` implicitly.
- 🟢 **OPS-2** [LOW] No structured logging or external error tracking (Sentry/Rollbar) — `console.log`/`console.error` only (`server/collab.js` throughout, `server.js:244-249`). Reasonable for the current scale; worth reconsidering if this moves beyond demo/reference status.

## 8. Tech-debt assessment

| Category | Effort to fix | Risk of not fixing | Compound effect |
|---|---|---|---|

## 9. Refactoring roadmap

```

Phase 4 (polish, ongoing):
