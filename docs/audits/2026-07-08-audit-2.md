## 1. Executive Summary

Das Tagging ist **architektonisch sauber** gelöst: ein **Standoff-Modell**
(Annotationen liegen *neben* dem Text, verankert über Zitat + Vorkommen — nie im
Markdown), gerendert als ProseMirror-**Decorations** (Dokument bleibt frei von
Tagging-Markup), synchronisiert über eine **Y.Array im selben Yjs-Dokument**, und
persistiert als edu-sharing-Keywords im Muster `"Name (Typ)"`. Die UI-Hygiene ist
gut (durchgängig `escapeHtml`, Fokus-Trap, ARIA), die Read-Back-Verifikation
deckt jetzt Markdown **und** Keywords ab, und ein cleveres Snapshot-Verfahren
verhindert Yjs-Duplikate bei Reconnect. Neue Tests decken die vorher ungetesteten
Logik- und DOM-Teile ab.

**Ein Finding wiegt schwer:** Die Keyword-Serialisierung interpretiert **jedes**
bestehende Repo-Keyword im Muster `"Wort (Zusatz)"` als Entität. Ein solches
Keyword, dessen `Wort` nicht wörtlich im Text steht, wird beim **nächsten
Speichern still und dauerhaft gelöscht** — ausgelöst durch eine *beliebige*
Textänderung, ohne dass die Redaktion Tagging überhaupt genutzt hat. In der
WLO-Metadatenpraxis sind Klammer-Disambiguierungen (`"Merkur (Planet)"`)
gebräuchlich → realer Verlust bestehender Produktionsmetadaten.

**Top-3:**
1. 🟠 **Silenter Keyword-Verlust** bei parenthesierten Keywords ohne Text-Vorkommen (Datenintegrität).
2. 🟡 **Basic-Auth-Passthrough ohne Rate-Limit** (Übernahme aus Audit 1, unverändert).
3. 🟡 **Plain-Keyword wird beim Laden zur Entität umgedeutet** (gleiche Wurzel wie #1, ohne Verlust, aber semantisch überraschend).

**Wichtigste Einzelmaßnahme:** Beim Laden markieren, **welche** Keywords
tatsächlich als Annotation übernommen wurden; beim Speichern nur diese ersetzen
und alle übrigen (auch parenthesierten) unangetastet durchreichen.

## 3. Findings

### F-T1 🟠 High — Silenter Verlust parenthesierter Keywords beim Speichern
```
File:    src/annotations.js
Lines:   155–163 (annotationsToKeywords), 133–147 (keywordsToAnnotations)
         + server/collab.js:131, 143 (Aufruf beim Speichern, Overwrite via saveKeywords)
Problem: annotationsToKeywords filtert ALLE Keywords im Muster "Name (Typ)" aus
         den `plain`-Keywords heraus und baut die Entitäts-Keywords ausschließlich
         aus den AKTUELLEN Annotationen neu. keywordsToAnnotations erzeugt beim
         Laden aber nur dann eine Annotation, wenn das Zitat WÖRTLICH im Text steht.
Ablauf:  Repo-Keyword "Merkur (Planet)", "Merkur" NICHT im Kompendiumtext →
         Laden: keine Annotation (Zitat nicht gefunden).
         Nutzer ändert IRGENDETWAS am Text → keywordsChanged=true →
         saveKeywords überschreibt die Liste OHNE "Merkur (Planet)".
Impact:  Bestehende redaktionelle Keywords im Klammer-Muster (in WLO für
         Disambiguierung üblich: "Golf (Sport)", "Merkur (Planet)") werden
         STILL und dauerhaft gelöscht — schon durch normale Textbearbeitung,
         ohne dass Tagging genutzt wurde. Kein Warnhinweis, Read-Back bestätigt
         sogar den (reduzierten) Stand als „gespeichert".
Fix:     Beim Laden die Menge der als Annotation ÜBERNOMMENEN Keywords festhalten
         (bzw. „consumed"-Marker). Beim Speichern nur genau diese durch die aus
         Annotationen abgeleiteten ersetzen; jedes andere Keyword (auch
         parenthesierte, die nicht als Entität geladen wurden) unverändert
         durchreichen. Alternativ Entitäts-Keywords eindeutig markieren
         (eigenes Präfix/Property) statt per Form-Heuristik zu erraten.
```

### F-T2 🟡 Medium — Basic-Auth-Passthrough umgeht das Login-Rate-Limit
```
File:    server.js (Routen /api/nodes/:id, /save, /autosave) + server/sessions.js
         (resolveAuthToken lässt "Basic …" durch)
Problem: Das Rate-Limit greift nur auf /api/login. Die Node-Routen akzeptieren
         rohe `Authorization: Basic <base64>` und reichen sie an edu-sharing
         weiter (getNodeInfo/checkWriteAccess) — ungedrosselt.
Impact:  Der Server wird zum unthrottled Credential-Guessing-Oracle gegen die
         edu-sharing-Konten (bereits im Audit 1 als Top-Risiko benannt, unverändert).
Fix:     Denselben Per-IP-Throttle auf alle Endpunkte legen, die einen rohen
         Basic-Header akzeptieren — oder Basic-Passthrough ganz entfernen und
         nur Session-Tokens (Bearer) zulassen.
Note:    Teilweise gemildert dadurch, dass edu-sharing die eigentliche
         Auth-Instanz ist; die App fügt aber einen bequemen ungedrosselten
         Proxy hinzu.
```

### F-T3 🟡 Low–Medium — Plain-Keyword im "(…)"-Muster wird zur Entität umgedeutet
```
File:    src/annotations.js:133–147 (keywordsToAnnotations)
Problem: Auch wenn NICHT verloren (Name steht im Text), wird ein rein
         redaktionelles Keyword wie "Golf (Sport)" beim Laden zur semantischen
         Entität — erscheint als Entity-Chip und farbige Text-Dekoration.
Impact:  Überraschende Umdeutung bestehender Metadaten; die Redaktion sieht
         plötzlich „Tags", die sie nie gesetzt hat. Gleiche Wurzel wie F-T1
         (Form-Heuristik statt expliziter Markierung).
Fix:     wie F-T1 — Entitäten explizit markieren statt aus der Klammer-Form zu raten.
```

### F-T4 🟢 Low — Decorations werden pro Tastendruck komplett neu berechnet
```
File:    src/annotation-extension.js:66–89, 118–121
Problem: buildDecorations/resolveToPmRanges bauen bei jedem docChanged den
         vollständigen Text-Index neu und suchen alle Zitate erneut
         (O(Annotationen × Vorkommen × Segmente) pro Tastendruck).
Impact:  Für typische Kompendiumtexte unkritisch; bei sehr großen, stark
         getaggten Dokumenten spürbare Editier-Latenz möglich.
Fix:     Erst bei tatsächlicher Textänderung neu auflösen (schon so) — zusätzlich
         den Text-Index cachen/inkrementell führen, falls Latenz auftritt.
```

### F-T5 🟢 Low — `md-collab-editor.js` mit 414 Zeilen über dem Richtwert
```
File:    src/md-collab-editor.js (414 Z.)
Problem: Über der 300-Zeilen-Schwelle. Annotationslogik ist bereits in den
         Controller ausgelagert; verbleibend: Editor-Setup, Save-Bar-Wiring,
         Awareness/Presence, Toolbar-Rendering in einer Klasse.
Impact:  Wartbarkeit; noch beherrschbar, aber Wachstumskandidat.
Fix:     Bei weiterem Wachstum Save-Bar-Wiring bzw. Presence in eigene Helfer
         auslagern (analog zur Annotation-Controller-Extraktion).
```

### F-T6 🟢 Low — AI-/Programmpfad prüft nicht auf blockübergreifende Zitate
```
File:    src/annotation-controller.js:55–63 (add) vs. 76 (openTagDialog prüft '\n')
Problem: Der interaktive Pfad lehnt Zitate mit '\n' ab (kein Block-Span); der
         programmatische add()-Pfad (AI-Einstieg) tut das nicht.
Impact:  Eine per API/AI gesetzte Annotation könnte über eine Blockgrenze
         hinweg verankert werden (im Text-Index sind Blöcke per '\n' getrennt).
Fix:     In add() denselben '\n'-Check anwenden bzw. in isValidQuote aufnehmen.
```

## 4. Was gut ist (keine Findings)

- **Standoff-Modell** („Zitate für die KI, Offsets für den Code") — Annotationen
  überleben kollaboratives Editieren, weil jeder Client gegen denselben Text
  auflöst; KI-Annotationen brauchen keine Positionsarithmetik.
- **Decorations statt Marks** — der Markdown-Export bleibt sauber; kein
  Tagging-Markup im gespeicherten Text.
- **Read-Back deckt jetzt Keywords mit ab** (`markdownOk && keywordsOk`,
  `sameList(..., unordered)`).
- **docSnapshots** gegen Yjs-Duplikate bei kurzem Reconnect — durchdachte Lösung
  eines echten CRDT-Fallstricks (Rebuild erzeugt strukturell neuen Y.Doc).
- **UI-Sicherheit/A11y:** `escapeHtml` konsequent bei allen innerHTML-Interpolationen,
  Presence/Chips per `textContent`, Fokus-Trap + Escape/Outside-Close + ARIA im Popup.
- **i18n** sauber getrennt: gespeicherte Typ-VALUES bleiben deutsch (Datenkonstanz),
  nur Labels übersetzt.
- **Testabdeckung** der Neuzugänge (Logik + jsdom-DOM + i18n) — schließt eine
  Lücke aus Audit 1.
